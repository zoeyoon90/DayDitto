# Pre-push 검증 + PR 본문 작성

push 전에 코드 품질을 검증하고, 통과하면 브랜치의 작업 내용을 논리적인 PR 본문으로 정리해 사용자에게 전달하는 워크플로우.

**절대 규칙: Claude는 절대 push하지 않는다. `git push`, `gh pr create` 등 원격에 반영되는 명령은 어떤 경우에도 실행하지 않는다. Claude의 역할은 검증과 PR 본문 작성까지이며, push는 사용자가 직접 한다.**

## 0단계: 사전 확인

1. `git branch --show-current`로 현재 브랜치 확인
   - `main`, `master`, `develop`이면 **즉시 중단**하고 사용자에게 알림 (보호 브랜치에 직접 push 방지)
2. `git status`로 커밋되지 않은 변경 확인
   - 미커밋 변경이 있으면 사용자에게 알리고 커밋할지 물어봄
3. 루트 `package.json`의 `scripts`를 읽어 실제 명령어 확인
   - 기본 가정: `pnpm lint`, `pnpm typecheck` (또는 `check-types`), `pnpm build`, `pnpm test`
   - 스크립트가 없는 항목(예: test 미구성)은 건너뛰되, 건너뛴 사실을 최종 보고에 명시

## 1단계: 검증 파이프라인

아래 순서대로 실행한다. 순서에 이유가 있다: 빠르고 값싼 검사를 먼저, 느린 검사를 나중에.

```
1. lint        (빠름, 스타일/규칙 위반)
2. typecheck   (타입 오류)
3. build       (실제 빌드 가능 여부)
4. test        (동작 검증, 가장 느림)
```

**각 단계 실패 시 (순서 엄수):**
1. 즉시 파이프라인 중단 (다음 단계 진행 금지)
2. **왜 실패했는지 설명한다** — 에러 메시지를 그대로 붙여넣는 게 아니라:
   - 원인 파일과 라인
   - 에러가 발생한 이유 (예: "apiFetch에서 제네릭 T를 반환하는데 호출부에서 타입 인자를 안 넘겨서 unknown으로 추론됨")
   - 어떻게 고칠 수 있는지 수정안 제시
3. **사용자 동의를 받은 후에만** 수정한다. 동의 없이 코드를 고치지 않는다
4. 수정 후 **해당 단계부터 파이프라인을 재실행**한다 (수정이 다른 검사를 깨뜨릴 수 있으므로 수정한 단계 → 이후 단계 전부 재검증)

**모두 통과 시:** 각 단계 결과를 한 줄씩 요약 (예: `✓ lint 통과 / ✓ typecheck 통과 / ✓ build 12.3s / ✓ test 24 passed`)

## 2단계: 변경사항 분석

PR 본문을 쓰기 전에 브랜치에서 무엇이 바뀌었는지 파악한다.

**base 브랜치 판단 (하드코딩 금지):**

```bash
if git show-ref --verify --quiet refs/heads/develop; then
  BASE=develop
elif git show-ref --verify --quiet refs/heads/main; then
  BASE=main
else
  BASE=master
fi
echo "base 브랜치: $BASE"
```

- `develop` 브랜치가 로컬에 존재하면 무조건 `develop`을 base로 사용한다 (git-flow 구조에서는 feature 브랜치가 develop에서 갈라져 나오므로, main과 비교하면 diff에 전체 프로젝트가 잡히는 오류가 생긴다)
- `develop`이 없으면 `main`, 그것도 없으면 `master`로 폴백한다
- 위 자동 판단이 실제 브랜치 구조와 맞지 않아 보이면(예: diff가 비정상적으로 크게 나오는 등) 추측하지 말고 **사용자에게 base 브랜치를 직접 질문**한다

**판단된 `$BASE`로 아래 명령 실행:**

```bash
git log $BASE..HEAD --oneline          # 커밋 목록
git diff $BASE...HEAD --stat           # 변경 파일 통계
git diff $BASE...HEAD                  # 실제 변경 내용
```

diff가 매우 크면 파일별로 나눠 읽는다. 분석하면서 다음을 파악한다:

- **무엇을** 추가/수정/삭제했는지 (기능 단위로 묶어서)
- **왜** 했는지 — 커밋 메시지, 코드 주석, 변경 패턴에서 의도를 추론
- **수정(리팩토링/버그픽스)이라면 이전 방식의 문제가 무엇이었는지**
- **버그 수정이 포함되어 있다면, diff만으로는 안 보이는 조사 과정(증상 → 원인 추적 → 해결)이 있는지 확인한다.** 커밋 메시지, 코드 주석, 대화 맥락에서 확인 가능하면 3단계의 트러블슈팅 섹션에 쓸 재료로 남겨둔다. 확인이 안 되면 추측하지 말고 사용자에게 직접 질문한다
- 의도가 diff만으로 불명확한 변경은 **추측으로 채우지 말고 사용자에게 직접 질문**

## 3단계: PR 본문 작성

아래 형식으로 작성한다. 핵심은 "무엇을"이 아니라 **"왜"** — 리뷰어가 diff를 열기 전에 맥락을 이해할 수 있어야 한다.

```markdown
## 개요
이 PR이 해결하는 문제 / 추가하는 기능을 2~3문장으로. 배경(왜 이 작업이 필요했는지) 포함.

## 변경사항
기능 단위로 묶어서 작성. 각 항목마다:
- **무엇을**: 변경 내용
- **왜**: 이 방식을 선택한 이유 (대안이 있었다면 왜 이걸 골랐는지)

### 예시
- `apiFetch` 래퍼 추가 (`lib/api.ts`)
  - 모든 API 호출에 credentials, 공통 헤더, 에러 처리가 반복되는 문제를 해결
  - axios 대신 fetch 기반으로 구현해 의존성 없이 유지

## 수정 사항 (기존 코드를 변경한 경우)
- **변경 전 → 변경 후**와 그 이유
- 예: `Credential: true` → `credentials: true` — 오타로 인해 CORS 옵션이 무시되던 버그 수정

## 트러블슈팅 (버그 수정이 포함된 경우에만 작성)
"수정 사항"이 변경 전/후의 diff 레벨이라면, 이 섹션은 그 위의 조사 과정을 담는다. diff만 봐서는 드러나지 않는, 원인을 추적하는 과정 자체가 리뷰어와 이후 비슷한 문제를 만날 사람에게 가치 있는 정보이기 때문이다.

- **증상**: 어떤 문제가 어떻게 관찰됐는지 (재현 조건 포함)
- **원인**: 근본 원인 — 조사 과정에서 확인한 사실 위주로. 표면적 증상과 실제 원인이 다른 경우 그 과정을 포함
- **해결**: 어떻게 고쳤는지, 왜 그 방법을 선택했는지 (다른 대안이 있었다면 왜 배제했는지)

### 예시
- **증상**: `DELETE /api/favorites/:id` 호출 시 404 발생. `GET /api/favorites`는 정상 동작
- **원인**: next.config.js의 리라이트 설정 때문에 동적 경로 요청만 NestJS(`/favorites/:id`, 존재하지 않는 경로)로 넘어가고 있었음. NestJS에는 `/favorite-expressions/:id`로 등록되어 있어 경로 불일치
- **해결**: 이미 Next.js route handler가 모든 `/api/*`를 처리하고 있어 리라이트 자체가 불필요했음 → 리라이트 규칙 삭제

버그 수정이 없는 PR(순수 기능 추가 등)이라면 이 섹션은 생략한다.

## 테스트 방법
리뷰어가 이 변경을 검증하는 방법 (실행 명령, 확인할 화면/엔드포인트)

## 리뷰 포인트
리뷰어가 특히 봐줬으면 하는 부분, 고민됐던 설계 결정
```

**작성 원칙:**
- 커밋 메시지를 그대로 나열하지 않는다. 커밋은 시간순이지만 PR 본문은 논리순으로 재구성한다
- diff에서 확인되지 않는 이유를 지어내지 않는다. 불명확하면 사용자에게 물어본다
- 트러블슈팅 섹션도 마찬가지로, 조사 과정이나 원인을 추측으로 채우지 않는다. 확인이 안 되면 사용자에게 직접 물어본다
- 트러블슈팅 섹션은 버그 수정이 포함된 PR에만 조건부로 넣는다. 모든 PR에 기계적으로 넣지 않는다
- 사소한 변경(포맷팅, 오타)은 한 줄로 묶는다
- 한국어로 작성한다

## 4단계: PR 본문 전달 (push는 하지 않음)

1. 작성한 PR 본문을 사용자에게 보여준다 (제목 + 본문, 그대로 복사해서 쓸 수 있는 형태로)
2. 수정 요청이 있으면 반영한다
3. **여기서 워크플로우 종료.** push와 PR 생성은 사용자가 직접 한다
4. 참고용으로 사용자가 실행할 명령만 안내한다 (실행하지 않는다):
   ```bash
   git push -u origin <브랜치명>
   ```

## 실패/예외 처리 요약

| 상황 | 행동 |
|------|------|
| push/PR 생성 요청 | Claude는 실행하지 않음. 명령어만 안내 |
| 보호 브랜치(main 등)에서 실행 | 즉시 중단, 사용자에게 알림 |
| 검증 실패 | 중단 → 실패 이유 설명 → 수정안 제시 → 동의 시 수정 → 해당 단계부터 재검증 |
| 미커밋 변경 존재 | 사용자에게 커밋 여부 질문 |
| base 브랜치 자동 판단이 실제 구조와 안 맞아 보임 | 추측하지 말고 사용자에게 질문 |
| 변경 의도 불명확 | 추측하지 말고 사용자에게 질문 |
| 버그 수정의 원인/조사 과정이 diff만으로 불명확 | 추측하지 말고 사용자에게 질문 |
| test 스크립트 없음 | 건너뛰고 최종 보고에 명시 |
