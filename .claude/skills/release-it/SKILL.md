# 릴리즈 워크플로우 (release-it)

`release-it`으로 버전을 올리기 전에 무슨 일이 일어날지 먼저 보여주고, 사용자 승인을 받은 뒤 진행하는 워크플로우.

**절대 규칙: Claude는 절대 실제 릴리즈를 실행하지 않는다. `pnpm release`(dry-run 제외), `git push`, `git tag` 등 원격에 반영되거나 버전을 실제로 변경하는 명령은 어떤 경우에도 직접 실행하지 않는다. Claude의 역할은 사전 점검과 dry-run 결과 설명까지이며, 실제 릴리즈 실행은 사용자가 직접 한다.**

## 0단계: 사전 확인

1. `git branch --show-current`로 현재 브랜치 확인
   - 릴리즈는 반드시 `main`(또는 프로젝트의 프로덕션 브랜치)에서 진행되어야 함. 다른 브랜치에 있으면 사용자에게 알리고 전환 여부 확인
2. `git status`로 미커밋 변경 확인 — 있으면 릴리즈 전에 정리하도록 안내
3. `git pull origin main`(또는 해당 브랜치)로 최신 상태인지 확인
4. `package.json`의 `scripts`에서 release 관련 명령어(`release`, `release:dry` 등) 확인

## 1단계: Dry-run 실행 및 결과 설명

```bash
pnpm release --dry-run
```

이 결과를 그대로 붙여넣지 말고, 다음을 명확히 정리해서 설명한다.

- 현재 버전 → 다음 버전으로 왜 그렇게 올라가는지 (patch/minor/major 판단 근거, 커밋 메시지 기반이라면 어떤 커밋이 근거인지)
- `CHANGELOG.md`에 어떤 항목들이 추가될 예정인지
- 어떤 git 태그가 생성될 예정인지
- **마지막 단계에 `main`으로의 직접 push가 포함되어 있는지 반드시 확인.** 브랜치 보호 규칙(Required PR 등)이 걸려있는 저장소라면 이 단계에서 실패할 수 있음을 미리 경고

## 2단계: 브랜치 보호 규칙 충돌 시 대응

dry-run 결과 push가 필요한데 main이 보호되어 있다면, 다음 옵션을 사용자에게 제시하고 선택하게 한다. 임의로 하나를 골라 진행하지 않는다.

1. 버전 bump 커밋을 별도 브랜치로 만들어 PR로 main에 반영 (다른 코드 변경과 동일한 흐름)
2. GitHub 저장소 Rulesets의 Bypass list에 본인 계정을 추가해 직접 push 허용
3. 버전 bump 커밋은 PR로 반영하고, git 태그만 merge 후 별도로 push

## 3단계: 사용자 승인 후 실제 실행 안내 (Claude는 실행하지 않음)

1. dry-run 결과와 위 옵션을 사용자에게 보여주고 진행 여부를 묻는다
2. 승인하면, 실제로 실행할 명령어를 안내만 한다 (Claude가 실행하지 않음):
   ```bash
   pnpm release
   ```
3. push/태그 생성이 필요한 후속 명령이 있다면 그것도 안내만 한다:
   ```bash
   git push --follow-tags origin main
   ```
4. **여기서 워크플로우 종료.** 실제 실행은 사용자가 직접 한다

## 실패/예외 처리 요약

| 상황 | 행동 |
|------|------|
| 실제 릴리즈 실행 요청 | Claude는 실행하지 않음. 명령어만 안내 |
| main이 아닌 브랜치에서 요청 | 브랜치 전환 필요 안내, 임의로 전환하지 않음 |
| 미커밋 변경 존재 | 사용자에게 정리 여부 질문 |
| dry-run 결과 push 단계가 브랜치 보호 규칙과 충돌 가능성 있음 | 2단계 옵션 제시, 사용자 선택 대기 |
| release 스크립트 없음 | 설정이 안 되어 있음을 알리고 release-it 설치/설정부터 안내 |
